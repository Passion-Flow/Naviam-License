# ADR 0001 - 项目边界

日期：2026-04-27
状态：已采纳

## 背景

需要明确 License 平台的范围，避免在第一版引入财务、客户自助、多租户等高复杂度能力。

## 决策

第一版包含：

- 厂商内部签发 / 续期 / 撤销 License。
- 在线 + 离线双通道激活。
- 单一超级管理员账号 + 强制 2FA。
- 哈希链审计 + 签名导出。
- Postgres + Redis 两个 Service。
- API + Web 两个镜像。
- Python SDK 离线校验。

第一版不包含：

- 按 feature 颗粒度签发。
- 多角色 RBAC。
- 客户自助门户。
- 计费 / 订阅 / 续费财务。
- 异步 worker / Gateway / Sandbox 镜像。
- KMS / HSM（接口预留）。
- 用量统计。

## 理由

- 安全是第一目标。范围越窄越可信。
- 厂商内部使用，不需要 SaaS 化的复杂度。
- 单一签发者降低了密钥管理与审计审批的复杂度。
- 离线优先要求架构简单可信；feature flag / RBAC 都增加误用面。
- 客户产品语言为 Python，SDK 与 Console 同语言降低维护成本。

## 影响

- 模块结构按 N 个产品设计但仅接入 1 个；扩展第二个产品零迁移。
- 不引入异步队列；通知 outbox 由同步主流程 + 重试承担。
- 私钥保管 A 方案足以满足第一版；KMS 在第二阶段评估。
- 未来引入 RBAC 时需要新增 ADR 与迁移计划。

## 复核

- 第一阶段实现完成后回顾本 ADR；如范围发生收窄或扩张，新增 ADR 并不删除本 ADR。
