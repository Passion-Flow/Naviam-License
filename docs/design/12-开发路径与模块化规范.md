# 12 - 开发路径与模块化规范

## 总原则

按「功能 + 模块」分阶段推进。每个阶段定义：

- 入口点（哪个模块 / 哪个文件开始）。
- 完成定义（什么标志该阶段闭环）。
- 验证方式（自动化测试 + 人工演练）。
- 不做（避免镜面拓宽）。

## 阶段一：项目骨架与安全基线（入口）

### 完成定义

- `apps/api/` Django 项目可启动，`/healthz` 与 `/readyz` 返回 200。
- `apps/api/config/settings/base.py` 落地全部安全默认值（见 `02-开发技术栈设计.md`）。
- `src/modules/security/` 提供 Ed25519 签名与公钥导出能力（A 方案）。
- 审计哈希链 genesis 记录写入。
- `apps/web/` 可登录到一个空白 Dashboard（仅显示用户信息）。

### 验证

- `pytest tests/integration/test_security_baseline.py`：会话 / CSRF / HSTS / CSP / Cookie 全部满足。
- 手动启动 Postgres + Redis Compose，`python manage.py migrate` + `runserver` + Web `pnpm dev` 可达。

### 不做

- 不实现 License 业务。
- 不实现客户 / 产品管理。

## 阶段二：账户与登录闭环

### 完成定义

- 登录 / 2FA / 备用码 / 强制改密 / 锁定 / 限速 / 审计写入完整。
- 默认管理员可登录、改密、绑定 TOTP。

### 验证

- 集成测试：`tests/integration/test_accounts.py` 覆盖正常登录、密码错误、锁定、2FA 错误、备用码使用、改密、注销、Session 失效。
- 安全测试：`tests/security/test_login_attack.py` 暴力破解、CSRF 缺失、Session 固定。

### 不做

- 不实现多角色。
- 不实现单点登录。

## 阶段三：客户与产品管理

### 完成定义

- 客户 CRUD（仅删除走软删 + 审计）。
- 产品种子（一条 `default`），UI 只读。

### 验证

- 集成测试：客户增删改查 + 审计可追溯。

### 不做

- 不实现产品多版本。
- 不实现批量导入。

## 阶段四：License 离线签发闭环（最关键）

### 完成定义

- `src/modules/activations/` 解码 Cloud ID（schema v1）。
- `src/modules/licenses/` 实现签发：从 Cloud ID + 客户 + 有效期 -> Ed25519 签名 -> Activation Code 输出。
- Console UI 完整支持「粘贴 Cloud ID -> 解析 -> 签发 -> 复制 Activation Code」。
- SDK 实现 `verify(license, cloud_id) -> Result`，可在无网络下运行。
- 完整审计写入。

### 验证

- 单元：`tests/unit/test_cloud_id_codec.py`、`test_license_codec.py`。
- 集成：`tests/integration/test_offline_issue.py` 跑完整流。
- 安全：`tests/security/test_signature_forgery.py`、`test_replay_protection.py`。

### 不做

- 不做在线激活。
- 不做心跳。

## 阶段五：续期 + 撤销

### 完成定义

- 续期：B2 同实例覆盖签发；产品端可识别新版本并替换。
- 撤销：C1 在线立即撤销 + 离线记录撤销 + 通知发送。
- UI 支持续期表单与撤销表单。

### 验证

- 集成测试：撤销后 SDK 校验返回 `revoked`。
- 安全测试：撤销后伪造 License 文件被拒绝。

### 不做

- 不做 C3 强制在线检查（高敏客户可选项）。

## 阶段六：在线激活 + 心跳

### 完成定义

- 在线签发：Console 主动 POST 客户实例激活回调（mTLS）。
- 心跳：SDK 在线模式定期上报；Console 记录最后心跳。
- 自动续期 B3：到期前自动拉取新 License。

### 验证

- 集成测试：mTLS、签名、回调失败重试、撤销下发。
- 端到端：模拟产品实例的 fake server 与 Console 互连。

### 不做

- 不做用量计量（与签发能力无关）。

## 阶段七：通知 + 审计导出

### 完成定义

- 通知通道：邮件 + Webhook（HMAC-SHA256 签名）。
- 重试 / 退避 / 持久化 outbox。
- 审计签名导出（zip = data.jsonl + chain.json + pubkey.pem + signature.bin）。

### 验证

- 集成测试：通知幂等、重试上限、签名校验。
- 安全测试：导出包篡改可证伪。

## 阶段八：私有化交付与监控

### 完成定义

- Helm chart `infra/helm/license/` 可在 Kubernetes 集群部署。
- Prometheus metrics + Grafana 模板。
- 离线交付包脚本（镜像 tar + chart + values + SBOM）。

### 验证

- 演练：在新集群从离线包部署 -> 首次登录 -> 强制改密 + 2FA -> 签发 -> 撤销 -> 备份 -> 还原。

## 模块化规范

- 每个模块（`src/modules/<m>/`）必须有：
  - `README.md`：边界 / 上下游 / 协作规则。
  - `services.py`：业务编排；视图层不允许跨模块直接读 model。
  - `tests/`：单元 + 集成。
- 跨模块通信：通过 `services.<action>` 函数调用，不允许跨模块直接 ORM。
- 公共契约只放 `src/contracts/`；`shared/` 只放纯工具（DTO、序列化、异常）。

## 不可妥协

- 安全测试不通过 -> 阶段不闭环。
- 审计哈希链可篡改 -> 阶段不闭环。
- 默认管理员未强制改密 / 绑 2FA -> 阶段不闭环。
- License 任何字段被串改后仍能通过 SDK 校验 -> 阶段不闭环。
