# 01 - 需求视角挖掘

## 用户

| 角色 | 描述 | 使用场景 |
|---|---|---|
| 厂商超级管理员 | 唯一签发者 | 登录 Console -> 创建客户 -> 签发 / 续期 / 撤销 License |
| 厂商运维 | 维护 Console 部署本身 | 备份 Postgres / Redis、轮换密钥、查看审计 |
| 客户实施 | 在客户现场部署产品 | 拿到 Activation Code -> 粘贴到产品 UI |
| 产品实例 | 被签发对象 | 启动时校验 License、定期心跳（如有网络）、过期处理 |
| 客户管理员 | 客户侧产品管理员 | 在产品 UI 复制 Cloud ID -> 联系厂商索取 Activation Code |

第一版仅厂商超级管理员一个 Console 用户。后续如增加角色，须 ADR。

## 场景

### S1：在线签发

1. 厂商管理员登录 Console（密码 + 2FA）。
2. 创建客户（公司名、联系人、联系方式、合规备注）。
3. 选择产品（第一版只有 1 个）+ 客户实例 + 有效期。
4. Console 直接调用产品实例公开的激活回调（HTTPS + 双向校验）下发 License。
5. 产品实例落地、回执，签发记录进入审计哈希链。

### S2：离线签发（Dify 风格）

1. 客户在产品 UI 看到 Cloud ID（含 product_id + instance_id + 公钥指纹 + 硬件指纹），点击复制。
2. 客户把 Cloud ID 通过任意安全通道发给厂商管理员。
3. 厂商管理员在 Console 粘贴 Cloud ID -> 选择产品 + 有效期 -> 签发 -> 得到 Activation Code。
4. 厂商把 Activation Code 通过任意安全通道发给客户。
5. 客户把 Activation Code 粘贴回产品 UI；产品本地用厂商公钥验签，验签通过即落地。

### S3：到期处理（A2 + 30 天宽限）

- 到期前 30 / 7 / 1 天：产品在 UI 与日志告警；如在线，主动拉取最新 License。
- 到期后：产品进入只读 + 30 天宽限期，仅允许导出 / 续期相关操作。
- 宽限期结束：产品停止业务功能，仅保留登录 + 续期入口。

### S4：续期

- B2：同实例下使用新 Activation Code 覆盖旧的（同一 Cloud ID）。
- B3：在线模式下，产品自动从 Console 拉取新 License。

### S5：撤销

- C1：在线模式下立即生效；产品下次心跳收到撤销，进入只读。
- 离线撤销：Console 记录「已撤销」，邮件 + Webhook 通知客户；产品端在到期日前仍可用，到期后自然失效。

## 痛点

- 现有产品没有标准签发渠道，存在「人工发 zip + 邮件」的脆弱流程。
- 无法证明某条 License 是「合法签发的」还是「客户私自伪造的」。
- 无法在离线场景下安全校验。
- 无法做被破解后的损害控制。

## 成功标准

第一版必须满足：

- 厂商可以在 Console 完成签发、续期、撤销，全部进入哈希链审计。
- 产品实例可以在无网络环境下，仅凭 SDK 内置公钥完成校验。
- 任何攻击产生的非法 License 不能通过 SDK 校验；任何对审计链的篡改可被检出。
- Console 默认管理员账号必须在首次登录时强制改密 + 绑定 2FA。
- 全部生产配置经过 `.env.example` 变量化。

## 风险

- 离线撤销不可达：物理限制，依赖短有效期 + 在线检查缓解。
- 私钥保管：A 方案（加密文件 + passphrase）安全性依赖运维流程；接口预留以便后续接 KMS / HSM。
- 客户侧 SDK 反编译：Python SDK 易被读，公钥可见但不影响安全；私钥永远不下发。
- 时间篡改：客户机改时钟绕过过期。第一版要求 SDK 校验 License 的 `not_before / not_after` + 启动时与构建时间双向比对。
