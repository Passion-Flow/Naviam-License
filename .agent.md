[License 业务模型 — 核心决策]
═══════════════════════════════════════════════════════════════

### 验证模式（License 文件内嵌 `mode` 字段；Verifier 同时支持三种）
- `offline`：纯离线验签，不回连
- `hybrid`：默认离线验签 + 周期心跳回连（推荐生产默认）
- `online`：每次启动调 LA API 实时验证，离线缓存 N 天

### License 颗粒度（License 文件内嵌 `scope` 字段；LA 同时支持三种）
- `customer_x_product`：per-customer × per-product（一张管一组合）
- `customer_bundle`：per-customer，一张含多产品 features
- `instance`：per-instance（每个部署实例一张）

### 签名算法（License 文件内嵌 `alg` 字段；签发与验证都支持三种）
- `ed25519`（默认推荐）
- `rsa2048` / `rsa4096`
- `sm2`（国密，信创场景）

### 绑定模式（License 文件内嵌 `binding` 字段；Verifier 按字段执行）
- `none`：不绑定（仅靠时间 + 吊销）
- `soft`：采集部署指纹 + 心跳上报（推荐）
- `hard`：签发时硬绑指纹，Verifier 启动严格比对

[私钥存储]
═══════════════════════════════════════════════════════════════

私钥存储走**混合模式**（实现优先级从上到下）：
1. **本地加密文件**（默认）：私钥用主口令加密落盘，主口令走环境变量 `FORGE_KEY_MASTER_PASSPHRASE`
2. **Object Storage**（可选）：私钥密文存到对象存储，便于多副本同步；客户场景按需选
3. **外部 KMS**（可选 / 未来）：Vault / AWS KMS / 阿里 KMS / 云商 KMS 适配

实现要点：
- 私钥服务启动时一次性加载到内存
- **绝不**写入日志、错误响应、API 响应
- 必须支持密钥轮换（生成新密钥 / 用新密钥签新发 / 旧密钥仅用于验旧 license）

[Verifier 设计]
═══════════════════════════════════════════════════════════════

### 支持语言（按优先级）
1. **Python**（必选 / 与后端默认栈一致）—— 骨架内唯一完整实现
2. **TypeScript / Node.js**（次选 / 占位）
3. **Go**（再次 / 占位）

### License 文件格式：`.forge`
- 实质是 tar 包，包含：
  - `payload.json`：业务字段（customer / product / expiry / features / mode / binding ...）
  - `signature.bin`：detached signature
  - `metadata.json`：签名算法 / 签名时间 / 签发者 / 公钥 ID
- 人可读，便于客户人工排障

### SDK 分发方式
- **项目选择**：可拷源码 / 私有 PyPI 或 npm / git submodule，由消费方项目自定
- Forge 仓库提供"参考实现"，**不**作为强制依赖

### 验签失败时行为
- **启动硬卡 + 定期复查**：
  - 启动时验签失败 → 进程 `exit 1`，错误清楚指引
  - 启动成功后，每 5 分钟（或 1 小时，项目可配）复查
  - 运行时 license 失效（过期 / 被吊销 / 心跳判定多环境占用）→ 广播业务层 + 进入"只读 / 部分禁用"模式

[镜像规划（设计阶段定型，遵守 04-Deployment HARD RULE）]
═══════════════════════════════════════════════════════════════

Forge 共交付 **4 个镜像**：

| 镜像名               | 运行时角色                       | 源代码目录                          | Dockerfile                                         |
|----------------------|----------------------------------|-------------------------------------|----------------------------------------------------|
| `forge-api`          | FastAPI HTTP 服务（uvicorn）     | `Project-source/forge-server/`      | `Project-source/forge-server/Dockerfile.api`       |
| `forge-worker`       | Celery worker（后台任务）         | `Project-source/forge-server/`      | `Project-source/forge-server/Dockerfile.worker`    |
| `forge-scheduler`    | Celery beat（定时调度）          | `Project-source/forge-server/`      | `Project-source/forge-server/Dockerfile.scheduler` |
| `forge-web`          | Admin UI 静态资源（nginx serve） | `Project-source/forge-admin/`       | `Project-source/forge-admin/Dockerfile`            |

**不算镜像的产物**：
- `forge-verifier-sdk`（Python / TS / Go）—— SDK 库，由消费方项目自行集成

**单源多镜像说明**：
- `forge-server/` 一份源码产出 3 个镜像（api / worker / scheduler）
- 三个 Dockerfile 共享 builder 阶段（安装依赖），仅 ENTRYPOINT 不同
- 镜像 tag 保持一致（`forge-api:v0.1.0` 与 `forge-worker:v0.1.0` 同一发版）

### Celery 在哪里跑
- `forge-api` **不**跑 Celery —— 只接 HTTP，把任务放进 broker（redis db 2）
- `forge-worker` 跑 `celery -A app.workers worker --loglevel=info`
- `forge-scheduler` 跑 `celery -A app.workers beat --loglevel=info`
- 三者共享同一份配置（同一份 `.env`、同一组 service 连接信息）

## [2026-05-13] forge-deploy 三交付模式独立子目录

**规则内容**：用户原话："我看 deploy 里面是不是还少了 docker，现在只有 helm 啊。"
落地：forge-deploy/ 下三种交付模式各占独立子目录：
- `forge-deploy/docker/`（交付模式 1：docker-compose + Dockerfiles + .env.example）
- `forge-deploy/gitlab/`（交付模式 2：.gitlab-ci.yml + pipelines/ + variables.md）
- `forge-deploy/helm/`（交付模式 3：Chart 完整结构）

**背景**：原设计 docker-compose.yml 与 .gitlab-ci.yml 是 forge-deploy/ 根级文件、仅 helm/ 为目录，视觉不对称。用户在 2026-05-13 发现并要求三者对称。
**适用范围**：本项目部署交付物结构。可作为新建项目的参考实践（但不写入全局，全局仍允许 file 或 dir 形式）。
